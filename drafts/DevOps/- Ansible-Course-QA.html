Ansible Course QA

雖然可以不需要去上課，這樣的技術，我以前就寫過類似的東西，也有一些經驗，不過去上課的目的除了技術，很重要的就是走出去看看，看看別人怎麼做同樣的事情。

以下是我的回覆：

Q: 當你從事相關的實務工作，涉及「組態管理系統」時，你最討厭的事情是什麼？ 寫下你的痛點 (pain)：抱怨 (complaints)、恐懼 (fears)。

  1. 開發人員對 “設定檔” 的設計沒概念，系統管理、部署人員可控性差。
  2. 設定檔沒有足夠的彈性，像是臨時性、常態性、等使用場景設計。
  3. 設定檔的設計不夠靈活，沒有繼承概念、註解等。
  4. 設定檔看不出系統 Overview
  5. 設定檔可讀性差，沒有註解說明
  6. 設定檔本身的 Bug 很多，像是設定參數的定義範圍。
  7. 沒有預留測試參數，系統依賴性太高，無法彈性調整。像是 Fake Storage (NFS, AWS S3, File System …)，可測性差。
  8. 要重建特定目的環境時，所有人都跳腳，或者覺得沒必要。然後繼續使用很髒的環境。
  9. 測試環境紊亂，各式各樣的 script 部署沒有標準程序
  10. 上線的環境，大部分都只是能動而已，設定檔沒有『重構』，裡面的資訊，像是名稱、描述都不清楚。
    • 測試階段：不知道有哪些設定檔，甚至無法動設定檔，可測性差，無法深度驗證
    • 因為設定檔紊亂，所以無法與自動化，與其他應用程式接軌困難
    • 維運階段啥都不能動，啥都不知道能不能動。。。
  11. 組態檔格式大亂鬥， properties, json, ini, yaml

如果有一天，當你學到「能夠解決上述問題的實務技術」後，你最想要做什麼事情？ 請列出 具體 想做的幾件事情。

精神：

  • 自動化他們。
  • 人管系統，系統管機器。

制度:

  1. 回去開發系統程式的時候，做可測性、可部署、可維運的東西。
  2. 影響開發團隊 (Dev & Test & Operator)，讓大家知道能設計出能彈性部署的系統架構，落實 Infra as Code、CI、CD 的概念。
  3. 落實快速部署『可測性』的環境，而不是生不出環境可以付現問題。
  4. 部署『 環境』不應該是只有某些人才會，而是開發團隊的 Base Skill，包含 Developer、QA、Operator
  5. 部署環境的成本是可以估算的、可控制的

Q: 當你做了上述事情後，你最想要達成什麼目標？期待自己有什麼轉變、別人有什麼轉變，或是更高的願景⋯⋯

  • 自己：對團隊更有影響力
  • 配合其他方法 (docker)，快速建立具體而微 prototype，同時也可以應應需求，快速調整系統架構，不用改扣
  • 遠大的願景：改變台灣軟體開發團隊的觀念，做世界級的軟體。
