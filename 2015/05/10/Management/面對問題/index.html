面對問題

跟朋友聊到流程的問題，聊到 這篇文章，整理一下我的想法。

其實不管用啥開發方法，water fall、Scrum、扛棒、XP … whatever …. 重點在於：

  1. 問題能夠及早發現, 即時處理 -> 預防甚於治療
  2. 讓所有的人 (不管是 developer / QA / operator / PM) 都能專注
  3. 專案狀態是在可以評估、控管的。。

如果這兩點做不到，那用啥方法、工具、系統都只是在盲從而已 …

忘了說關於第一點，遇到問題要去面對，而不是忽略，最近發現一些公司老版都有同樣的問題，難怪台灣的公司等級都無法有國際級的水準，永遠只停留在代工、生產、製造、等著被併購。

只顧眼前，永遠在處理很急的事情，沒有把視野、戰場放遠，放大。一家公司的一級主管一直都在處理這樣的問題，然後老闆只想把這家公司以高價賣出、東西趕快上市、不管好或壞，期望總有一天被人買走，會變成 “be part of XXX” ，然後自己從中獲利 …. 剩下的他就不管了 …

為什麼不會想讓自己變成下一個 Apple、Google、IBM、Microsoft … 即使者是小米、掏寶 … (我不覺得掏寶或小米真的比較厲害 …)

面對問題，有幾種解決，第一種：處理眼前的過一天算一天、第二種：分析後，建立適當且可以客制、設定的方法、流程，然後解決自己的問題的同時也創造附加價值。

目前遇到的大部分都還在第一個，或是在第一個無窮迴圈。不過在 github 上不難發現，很多工具的目的，都是第二種，更甚者，像 Amazon 那樣，把整個銷售的工具，變成另一種價值 (AWS)。

這些問題歸納有以下幾點：

  1. 經驗沒有繼承，一直在重新造輪子
  2. 沒有輕重緩急，特別是大老闆的信念不穩定，員工整天在瞎忙
  3. 老闆的視野太淺短，員工不敢說真話。

現在我還沒遇到這樣的老闆 …. 大部份處理眼前的事情，沒空理我，不曉得是我太理想，不知道天高地厚，還是他們真如我所述？
